1. Введение в C# и платформу .NET
1.1. Обзор платформы .NET

История и эволюция:

  • .NET Framework:
    • Исходная реализация платформы.
    • Глубокая интеграция с операционной системой, ограниченная кроссплатформенность.
    • Предназначена исключительно для Windows.
    • Основные версии: от 1.0 до 4.8.1.
  • .NET Core:
    • Полная переработка платформы с ориентацией на кроссплатформенность (Windows, Linux, macOS).
    • Модульная архитектура, допускающая развёртывание только необходимых компонентов.
    • Высокая производительность.
    • Основные версии: от 1.0 до 3.1.
  • .NET (начиная с версии 5):
    • Развитие платформы .NET Core в виде единой современной платформы .NET.
    • .NET Framework остался отдельной платформой, поддерживаемой только на Windows.
    • Общий runtime и единая экосистема для большинства типов приложений.
    • Начиная с .NET 6 Microsoft придерживается стратегии, при которой версии с чётными номерами обычно получают статус LTS (Long Term Support) с продолжительным периодом сопровождения.

Составляющие платформы .NET:

  • CLR (Common Language Runtime) — среда выполнения управляемого кода. Обеспечивает загрузку сборок, JIT-компиляцию, управление памятью (сборку мусора), проверку типов и обработку исключений.
    • В .NET для настольных и серверных платформ (Windows, Linux, macOS) среда выполнения основана на CoreCLR. Для мобильных платформ (Android, iOS) и Mac Catalyst в .NET MAUI используется среда выполнения Mono.
  • BCL (Base Class Library) — базовая библиотека классов. Включает фундаментальные типы данных, коллекции, средства работы с файловой системой, сетью, потоками, текстом и XML. В состав платформы также входят современные API сериализации JSON (System.Text.Json) и другие стандартные библиотеки.
  • SDK (Software Development Kit) — комплект средств разработки. Включает компилятор Roslyn, интерфейс командной строки dotnet, шаблоны проектов и утилиты сборки.
  • Рабочие нагрузки (Workloads) — специализированные наборы компонентов, устанавливаемые поверх SDK или через установщик Visual Studio для поддержки конкретных типов приложений. В интерфейсе Visual Studio они представлены следующими типовыми наборами:
    • ASP.NET and Web Development — создание веб-приложений, API и одностраничных приложений (Blazor, React, Angular).
    • .NET Desktop Development — разработка настольных приложений Windows с использованием WPF и Windows Forms.
    • Windows App SDK (WinUI) — разработка современных настольных приложений Windows с использованием WinUI 3.
    • .NET Multi-platform App UI (MAUI) — создание кроссплатформенных приложений для Android, iOS, macOS и Windows из единой кодовой базы.
    • Azure Development — инструменты для создания приложений, развёртываемых в облаке Azure, и работы с рабочими процессами.
    • Game Development with Unity — средства разработки игр на языке C#.
  • При использовании командной строки (CLI) для управления рабочими нагрузками применяются идентификаторы рабочих нагрузок (workload IDs) (например, dotnet workload install maui, dotnet workload install wasm-tools). Базовые шаблоны SDK позволяют создавать консольные приложения и библиотеки классов без установки дополнительных workloads.

Управляемый код и среда CLR:

  • Управляемый код (Managed Code) — код, выполняемый под контролем среды CLR.
  • Процесс компиляции и выполнения:
    • Исходный код на языке C# компилируется в промежуточный язык — IL (также обозначается как CIL или MSIL) — платформонезависимый набор инструкций.
    • IL-код сохраняется в сборке (файле с расширением .exe или .dll) совместно с метаданными, описывающими типы и их члены.
    • При запуске приложения CLR загружает сборку и применяет JIT-компилятор для преобразования IL в машинный код, специфичный для текущей архитектуры процессора (x86, x64, ARM).
    • Обычно JIT-компиляция выполняется при первом использовании метода.
    • Сгенерированный машинный код хранится в памяти процесса во время выполнения приложения.
  • JIT-компиляция (Just-In-Time):
    • Преобразование IL в машинный код непосредственно перед выполнением метода.
    • Преимущества: переносимость сборок, возможность runtime-оптимизаций под конкретную архитектуру процессора.
    • Недостатки: незначительная задержка при первом вызове метода, расход памяти на хранение скомпилированного кода.
  • AOT-компиляция (Ahead-Of-Time):
    • Основная часть IL-кода заранее компилируется в машинный код на этапе публикации приложения.
    • Режим Native AOT доступен начиная с .NET 7.
    • Преимущества: сокращённое время запуска, сниженное потребление памяти, возможность выполнения на системах без отдельно установленного .NET Runtime, поскольку необходимые компоненты среды выполнения включаются в публикацию.
    • Недостатки: увеличение размера исполняемого файла. Некоторые JIT-оптимизации времени выполнения недоступны, поскольку код компилируется заранее, ограничения на использование рефлексии и динамической генерации кода.

Сборка мусора (Garbage Collection, GC):

  • Назначение — автоматическое освобождение памяти, занятой объектами, которые больше не достижимы из корневых ссылок приложения (GC Roots).
  • Управляемая куча (Managed Heap) используется главным образом для размещения объектов ссылочных типов и других данных, требующих динамического выделения памяти (упакованные значимые типы, массивы, строки и т.д.).
  • Поколения (Generations):
    • Поколение 0 — новые объекты. Сборка выполняется наиболее часто.
    • Поколение 1 — объекты, пережившие одну сборку поколения 0.
    • Поколение 2 — долгоживущие объекты. Сборка выполняется редко.
  • Крупные объекты дополнительно размещаются в Large Object Heap (LOH).
  • Принцип работы:
    • GC запускается автоматически CLR на основе эвристик (набора гибких правил) использования памяти.
    • Некоторые фазы GC могут временно приостанавливать управляемые потоки.
    • Освобождает память, занятую недостижимыми объектами.
    • Может выполнять уплотнение кучи для уменьшения фрагментации памяти (не для всех поколений и не во всех режимах).
  • Финализаторы (деструкторы):
    • Методы вида ~ClassName(), вызываемые сборщиком мусора перед окончательным освобождением памяти объекта. Финализатор может выполняться значительно позже момента потери всех ссылок на объект.
    • Обычно используются как резервный механизм освобождения неуправляемых ресурсов.
    • Использование без крайней необходимости не рекомендуется.
    • Предпочтительным является применение интерфейса IDisposable и конструкции using.

Языки платформы .NET (Common Language Infrastructure, CLI):

  • C# — основной язык платформы. Объектно-ориентированный, со строгой статической типизацией.
  • F# — мультипарадигмальный язык с акцентом на функциональное программирование.
  • Visual Basic .NET — объектно-ориентированный язык с синтаксисом, приближенным к естественному английскому.
  • Совместимость языков обеспечивается общей системой типов CTS (Common Type System) и инфраструктурой CLI. Большинство языков платформы используют типы BCL. Допускается вызов кода, написанного на одном языке, из другого.

Пример межъязыкового взаимодействия (C# и Visual Basic .NET):

  • Класс библиотеки на C#:
    КодC#
    namespace Utilities
{
    public class Calculator
    {
        public int Add(int a, int b) => a + b;
    }
}
  • Код на Visual Basic .NET, использующий класс из C#:
    КодVB.NET
    Imports System
Imports Utilities
Module Program
    Sub Main()
        Dim calc As New Calculator()
        Console.WriteLine(calc.Add(2, 3))
    End Sub
End Module
1.2. Установка и инструменты разработки

Установка .NET SDK:

  • Для начала разработки на языке C# с использованием платформы .NET требуется установка .NET SDK (Software Development Kit). SDK включает компилятор, инструменты сборки, шаблоны проектов, CLI и соответствующую версию .NET Runtime.
  • Дистрибутивы .NET SDK доступны для загрузки с официального веб-сайта dotnet.microsoft.com. Платформа поддерживает операционные системы Windows, Linux и macOS. Предоставляются установщики для графической установки, а также скрипты и пакеты для менеджеров пакетов операционных систем (APT, DNF/YUM, Homebrew).
  • Версии .NET SDK классифицируются по типу поддержки:
    • LTS (Long Term Support) — версии с долгосрочной поддержкой продолжительностью три года. Рекомендуются для промышленной эксплуатации и проектов, требующих стабильности.
    • STS (Standard Term Support) — версии со стандартным сроком поддержки (начиная с .NET 9 — два года). Содержат более новые возможности языка и платформы.
  • Допускается параллельная установка нескольких версий .NET SDK и соответствующих сред выполнения на одной рабочей станции. TargetFramework задаёт целевую платформу. Версия SDK дополнительно может фиксироваться через файл global.json.
  • Проверка корректности установки и получение сведений о текущем окружении выполняется командой:
    Терминалbash
    dotnet --info
    • Данная команда выводит информацию об установленных версиях SDK и runtime, путях их расположения, а также сведения об операционной системе и архитектуре процессора.

Среды разработки:

  • Для создания, редактирования и отладки исходного кода на C# могут использоваться различные интегрированные среды разработки (IDE) и редакторы кода. Основными вариантами являются:
    • Microsoft Visual Studio (Windows) — полнофункциональная среда разработки, предоставляющая средства визуального проектирования пользовательских интерфейсов (WPF, Windows Forms), расширенные инструменты отладки, профилирования и интеграции с облачными службами Azure. Распространяется в редакциях Community (бесплатная для индивидуальных разработчиков и небольших команд), Professional и Enterprise.
    • Visual Studio Code (кроссплатформенный) — легковесный редактор исходного кода с расширяемой архитектурой. Для комфортной работы с C# в VS Code рекомендуется установить расширения C# и C# Dev Kit, которые вместе обеспечивают поддержку языка (IntelliSense, навигация), отладку, управление решениями, шаблоны и инструменты тестирования. Часто используется для веб-разработки, облачных приложений и микросервисной архитектуры, но применим и для других типов проектов (десктоп, библиотеки классов, консольные приложения).
    • JetBrains Rider (кроссплатформенный) — коммерческая среда разработки, построенная на платформе IntelliJ и интегрирующая механизмы анализа кода ReSharper. Отличается высокой производительностью при работе с крупными решениями и расширенными возможностями статического анализа и рефакторинга.

Структура проекта .NET:

  • NuGet — стандартный менеджер пакетов для платформы .NET, используемый для добавления, восстановления и публикации внешних библиотек.
  • Проект .NET представляет собой совокупность файлов исходного кода, ресурсов и метаданных, организованных для компиляции в единый выходной файл (исполняемое приложение или библиотеку). Проект описывается файлом с расширением .csproj в формате XML. Основные элементы описания проекта включают:
    • OutputType — определяет тип выходного файла: Exe для исполняемого приложения или Library для библиотеки классов.
    • TargetFramework задаёт целевую версию .NET, для которой компилируется приложение.
    • ImplicitUsings — включает автоматическое добавление глобальных директив using для часто используемых пространств имён.
    • Nullable — активирует контекст анализа допустимости значений null для ссылочных типов.
  • Пример минимального файла проекта (.csproj):
    .csprojXML
    <Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <ImplicitUsings>enable</ImplicitUsings>
    <Nullable>enable</Nullable>
  </PropertyGroup>

</Project>
  • Совокупность проектов может быть объединена в решение (Solution). Решение описывается файлом с расширением .sln и содержит ссылки на входящие в него проекты, а также конфигурации сборки.
  • В рамках проекта зависимости подразделяются на три категории:
    • Framework References — ссылки на общие компоненты (shared frameworks), предоставляемые целевой платформой (например, Microsoft.AspNetCore.App, Microsoft.WindowsDesktop.App).
    • Packages — внешние библиотеки, полученные через менеджер пакетов NuGet.
    • Projects — ссылки на другие проекты, входящие в состав того же решения.
  • Файловая структура типового проекта включает следующие каталоги:
    • bin/ — каталог для размещения скомпилированных выходных файлов. Внутри создаются подкаталоги по конфигурациям (Debug/ и Release/), а внутри них — подкаталог целевой платформы (например, net8.0). Полный путь может выглядеть как bin/Debug/net10.0/.
    • obj/ — каталог для временных объектных файлов, создаваемых в процессе компиляции. Структура аналогична: obj/Debug/net10.0/.
  • Иерархия решения и проектов:
    Структура
    ИмяРешения.sln                   # Решение
│
├── ИмяПроекта.App/              # Основной проект
│   ├── ИмяПроекта.App.csproj    # Настройки проекта
│   ├── Program.cs               # Точка входа
│   ├── ИмяТипа1.cs              # Файл с описанием класса
│   └── ИмяПапки/                # Папка (подпространство имен)
│       └── ИмяТипа2.cs
│
└── ИмяПроекта.Tests/            # Проект с модульными тестами
    ├── ИмяПроекта.Tests.csproj
    └── UnitTests.cs

Создание и запуск консольного приложения:

  • Создание нового проекта консольного приложения осуществляется командой:
    Терминалbash
    dotnet new console -n ИмяПроекта
  • После создания проекта для его сборки и запуска используется команда:
    Терминалbash
    dotnet run
    • Данная команда автоматически выполняет восстановление зависимостей, компиляцию исходного кода и запуск полученного исполняемого файла.

Управление конфигурациями сборки:

  • Проекты .NET поддерживают две стандартные конфигурации сборки: Debug и Release.
    • Debug — конфигурация, предназначенная для этапа разработки и отладки. Оптимизация кода отключена, генерируются полные символы отладки (PDB-файлы), определены символы условной компиляции DEBUG и TRACE.
    • Release — конфигурация, предназначенная для финального развёртывания. Включена оптимизация кода. Наличие и объём отладочной информации определяются параметрами проекта; Release-сборки также могут содержать PDB-файлы. Автоматически определён символ TRACE; символ RELEASE по умолчанию не создаётся и при необходимости добавляется вручную через DefineConstants.
  • Выбор конфигурации при сборке из командной строки задаётся параметром -c или --configuration:
    Терминалbash
    dotnet build -c Release
  • Публикация приложения для развёртывания в целевой среде выполняется командой:
    Терминалbash
    dotnet publish -c Release -o ./publish
    • Параметр -o (или --output) указывает каталог, в который будут помещены файлы, готовые к развёртыванию.

Восстановление зависимостей:

  • Зависимости проекта, указанные в файле .csproj и описанные в конфигурационных файлах NuGet, восстанавливаются автоматически при выполнении команд build, run и publish. Явный вызов восстановления может быть выполнен командой:
    Терминалbash
    dotnet restore
    • Данная команда загружает все необходимые пакеты из настроенных источников (по умолчанию — nuget.org) в локальный кэш пакетов и обеспечивает их доступность для последующей компиляции.
1.3. Базовые понятия

Компиляция в промежуточный язык (IL):

  • Исходный код на языке C# компилируется не напрямую в машинный код, а в платформонезависимый промежуточный язык — IL (также обозначается как CIL или MSIL).
  • IL-код сохраняется в файлах сборок (расширение .exe или .dll) совместно с метаданными, описывающими типы, члены типов и внешние зависимости.
  • IL является платформонезависимым. Возможность запуска конкретной сборки на разных ОС зависит от используемых API и зависимостей (например, вызовы нативных методов через P/Invoke или привязка к платформенно-специфичным библиотекам могут ограничить переносимость).

Цепочка преобразования кода: Исходный код на C# -> (Roslyn) -> IL -> (JIT в составе CoreCLR) -> Машинный код

JIT-компиляция (Just-In-Time):

  • Принцип: при запуске приложения среда выполнения (CLR) вызывает JIT-компилятор. Обычно метод компилируется при первом обращении к нему, если для данного метода не используется предварительно сгенерированный машинный код (например, через ReadyToRun) или не применяется многоуровневая компиляция (Tiered Compilation).
  • Особенности:
    • Компилируются только те методы, которые фактически вызываются в ходе работы приложения.
    • Результат JIT-компиляции кэшируется в оперативной памяти на всё время существования процесса.
    • Компилятор способен применять оптимизации, учитывающие архитектуру текущего процессора.
  • Преимущества: переносимость сборок, адаптивная оптимизация.
  • Недостатки: незначительная задержка при первом вызове метода.

AOT-компиляция (Ahead-Of-Time):

  • Принцип: основная часть IL-кода заранее компилируется в машинный код на этапе публикации приложения, до его запуска.
  • Способы AOT в .NET:
    • Native AOT (доступен начиная с .NET 7) — компиляция основной части IL-кода в самодостаточный исполняемый файл, не требующий отдельно установленного .NET Runtime, поскольку необходимые компоненты среды выполнения включаются в публикацию.
    • ReadyToRun (R2R) — частичная предварительная компиляция (partial AOT) IL-кода для ускорения запуска приложения; JIT-компилятор при этом остаётся активным и может перекомпилировать методы.
  • Преимущества: сокращение времени запуска благодаря предварительной компиляции кода. Для Native AOT дополнительно характерны сниженное потребление оперативной памяти и возможность выполнения без отдельно установленного .NET Runtime (необходимые компоненты среды выполнения включаются в публикацию).
  • Недостатки: увеличение размера выходного файла, отсутствие некоторых оптимизаций времени выполнения, ограничения на использование рефлексии и динамической генерации кода.

Сборка мусора (Garbage Collection, GC):

  • Назначение — автоматическое освобождение памяти, занятой объектами, которые больше не достижимы из корневых ссылок приложения.
  • Управляемая куча (Managed Heap) — основная область памяти для размещения объектов ссылочных типов и других данных, требующих динамического выделения памяти (упакованные значимые типы, массивы, строки и др.).
  • Поколения (Generations):
    • Поколение 0 — новые объекты; сборка выполняется наиболее часто и быстро.
    • Поколение 1 — объекты, пережившие одну сборку поколения 0.
    • Поколение 2 — объекты, длительное время остающиеся достижимыми (например, долгоживущие экземпляры, существующие на протяжении всего времени работы приложения).
  • Режимы работы GC:
    • Workstation GC — предназначен для клиентских приложений, ориентирован на минимизацию пауз.
    • Server GC — предназначен для серверных приложений с высокой нагрузкой, использует многопоточную сборку для повышения пропускной способности.
  • Ключевые методы:
    • GC.Collect() — принудительный запуск сборки мусора (применение в пользовательском коде не рекомендуется).
    • GC.SuppressFinalize() — предотвращает вызов финализатора для указанного объекта, удаляя его из очереди финализации (если финализатор определён). Это позволяет сборщику мусора освободить память без дополнительной задержки на финализацию. Если финализатора нет, вызов метода не имеет эффекта.

Режимы сборки Debug и Release:

  • Debug (отладочная конфигурация):
    • Оптимизация кода отключена (<Optimize>false</Optimize>).
    • Часть оптимизаций JIT ограничивается для повышения качества отладки.
    • Генерируются полные файлы символов отладки (PDB), содержащие имена переменных и соответствие инструкций строкам исходного кода.
    • Определены символы условной компиляции DEBUG и TRACE.
  • Release (релизная конфигурация):
    • Оптимизация кода включена (<Optimize>true</Optimize>).
    • JIT-компилятор и этапы публикации (например, trimming, AOT) могут дополнительно сокращать и оптимизировать код.
    • Наличие и объём отладочной информации определяется параметрами проекта; Release-сборки также могут содержать PDB-файлы.
    • В конфигурации Release обычно определён символ TRACE.
    • Символ RELEASE не создаётся автоматически и должен быть добавлен вручную через DefineConstants при необходимости.
  • Переключение режимов:
    • Через интерфейс среды разработки (Visual Studio, Rider).
    • Через параметр командной строки: dotnet build -c Release.
    • Указанием в файле .csproj: <Configuration>Release</Configuration>.

Выбор конфигурации в зависимости от этапа:

  • Разработка и отладка: используется конфигурация Debug.
    • Позволяет устанавливать точки останова, выполнять пошаговую трассировку, просматривать значения локальных переменных и полей объектов.
    • Поддерживается изменение кода во время остановки отладчика (Edit & Continue).
    • Стек вызовов исключений содержит номера строк исходного кода.
  • Тестирование производительности и профилирование: допускается использование конфигурации Release с включённой генерацией переносимых символов отладки (<DebugType>portable</DebugType>). Это позволяет профилировать оптимизированный код.
  • Финальная сборка и развёртывание: используется конфигурация Release.
    • Обычно обеспечивает более эффективный машинный код и может уменьшать объём публикуемых артефактов в зависимости от настроек публикации.
    • Повышает скорость выполнения критических участков благодаря оптимизациям JIT и компилятора C#.

Условная компиляция:

  • Директивы препроцессора #if DEBUG / #if RELEASE позволяют включать или исключать блоки кода в зависимости от текущей конфигурации сборки:
    КодC#
    #if DEBUG
// Код, включаемый только в отладочной конфигурации
#endif

#if !DEBUG
// Код, включаемый только в релизной конфигурации
#endif
  • Если необходим именно символ RELEASE, его можно добавить вручную в файл проекта (.csproj):
    КодXML
    <PropertyGroup Condition="'$(Configuration)' == 'Release'">
  <DefineConstants>RELEASE</DefineConstants>
</PropertyGroup>
    После определения можно будет использовать #if RELEASE.
  • Атрибут [Conditional("DEBUG")] исключает вызовы метода из компилируемого кода, если символ DEBUG не определён (сам метод остаётся в сборке):
    КодC#
    [Conditional("DEBUG")]
static void DebugMethod()
{
   // Вызовы метода будут исключены из скомпилированного кода при отсутствии символа DEBUG
}

Отладка Release-сборки:

  • При возникновении ошибки в релизной версии приложения возможно подключение PDB-файлов, сгенерированных на этапе сборки.
  • В среде Visual Studio для просмотра полного стека вызовов может потребоваться отключение режима «Just My Code» и включение отладки нативного кода.
  • Конфигурация <DebugType>portable</DebugType> в сочетании с <Optimize>true</Optimize> обеспечивает компромисс: символы отладки доступны, но код остаётся оптимизированным.

Влияние на производительность и память:

  • Конфигурация Debug:
    • Более высокое потребление оперативной памяти (хранение отладочных данных, сниженное переиспользование регистров).
    • JIT-компилятор не применяет агрессивное встраивание методов, что упрощает анализ стека вызовов при отладке.
  • Конфигурация Release:
    • JIT-компилятор может выполнять встраивание методов (в том числе аксессоров свойств), удалять проверки границ массивов в безопасных случаях.
    • Различные этапы оптимизации (JIT-компиляция, IL trimming, AOT-компиляция) могут удалять недостижимый или неиспользуемый код и выполнять свёртку констант. Компилятор C# выдаёт предупреждения о недостижимом коде.

Практические рекомендации:

  • Не рекомендуется развёртывать сборки, собранные в конфигурации Debug, в промышленной среде. Это приводит к снижению производительности и потенциальному раскрытию внутренней логики приложения через отладочные сообщения.
  • Для проверки инвариантов в процессе разработки рекомендуется использовать Debug.Assert(). При отсутствии символа DEBUG вызовы Debug.Assert не включаются в результирующий код (сам метод остаётся в сборке, но его вызовы удаляются).
  • Для условного логирования вместо директив #if DEBUG предпочтительнее применять интерфейс ILogger с настройкой фильтрации по уровню важности сообщений.
  • При публикации однокомпонентного (single-file) приложения или приложения в режиме Native AOT символы отладки могут быть вынесены в отдельный PDB-файл либо исключены полностью.
1.4. Инструменты командной строки (dotnet CLI)

Инструменты командной строки (dotnet CLI):

  • dotnet CLI — кроссплатформенная утилита командной строки для разработки, сборки, запуска и публикации приложений .NET.
  • Входит в состав .NET SDK.
  • Позволяет выполнять все основные операции без использования интегрированной среды разработки (достаточно редактора кода и терминала).

Структура команд CLI:

  • Драйвер (dotnet) — предоставляет команды для работы с проектами .NET или запуска приложения .NET.
  • Команда — выполняет конкретное действие.
  • Опции — параметры, модифицирующие поведение команды.
  • Аргументы — значения, передаваемые команде или опциям.
  • Общий синтаксис:
    Терминалbash
    dotnet <COMMAND> [options] [arguments]

Основные команды:

  • Создание проекта, файла конфигурации или решения на основе указанного шаблона:
    Терминалbash
    dotnet new <TEMPLATE> [options]
  • Компиляция проектов .NET и их зависимостей в выходные файлы (сборки):
    Терминалbash
    dotnet build [<PROJECT>|<SOLUTION>] [-c|--configuration] [-o|--output]
  • Автоматическая сборка и запуск приложения (без необходимости отдельно вызывать dotnet build). Аргументы после -- передаются запускаемому приложению:
    Терминалbash
    dotnet run [-c|--configuration] [-f|--framework] [-- <args>]
  • Выполнение модульных тестов (начиная с .NET 10 SDK поддерживается выбор между VSTest и Microsoft Testing Platform; в более ранних версиях используется VSTest):
    Терминалbash
    dotnet test <PROJECT> [command-options] [arguments]
  • Публикация приложения и его зависимостей в каталог для последующего развёртывания:
    Терминалbash
    dotnet publish [<PROJECT>|<SOLUTION>] [-c|--configuration] [-r|--runtime] [-o|--output]
  • Добавление ссылки на пакет NuGet в проект (начиная с .NET 10 рекомендуется использовать dotnet package add; форма dotnet add package сохраняется для совместимости):
    Терминалbash
    dotnet package add <PACKAGE_NAME> [OPTIONS]
dotnet add [PROJECT] package [PACKAGE_NAME] [OPTIONS]

Общие флаги (применимы к большинству команд):

  • -c|--configuration — выбор конфигурации сборки: Debug или Release (по умолчанию Debug).
  • -o|--output — каталог для размещения выходных файлов.
  • -f|--framework — целевая версия платформы .NET (например, net8.0).
  • --no-restore — отказ от автоматического восстановления зависимостей.
  • -v|--verbosity — уровень детализации вывода: quiet, minimal, normal, detailed, diagnostic.

Местоположение установленных пакетов и инструментов:

  • Кэш NuGet:
    • ~/.nuget/packages (Linux/macOS)
    • %USERPROFILE%\.nuget\packages (Windows)
  • Глобальные инструменты .NET:
    • ~/.dotnet/tools (Linux/macOS)
    • %USERPROFILE%\.dotnet\tools (Windows)
  • Просмотр сведений о текущей установке:
    Терминалbash
    dotnet --info

Справка по командам:

  • Команда dotnet help <COMMAND> поддерживается для совместимости, однако чаще используется форма dotnet <COMMAND> --help для вывода справки в терминале:
    Терминалbash
    dotnet <COMMAND> -h|--help
1.5. Анализаторы кода и форматирование (.editorconfig)

Назначение .editorconfig:

  • Файл конфигурации, задающий единые правила оформления кода и поведения анализаторов для всех участников проекта.
  • Размещается в корне репозитория или каталоге проекта.
  • Настройки из .editorconfig имеют приоритет над личными параметрами среды разработки.
  • Поддерживается средами Visual Studio, Visual Studio Code, JetBrains Rider и большинством редакторов кода.

Базовый пример .editorconfig:

  • Файл .editorconfig с типовыми настройками для проектов на C#:
    Код.editorconfig
    root = true

[*]
indent_style = space
indent_size = 4
charset = utf-8
end_of_line = lf
trim_trailing_whitespace = true
insert_final_newline = true

[*.cs]
dotnet_diagnostic.CA1805.severity = warning
dotnet_diagnostic.IDE0055.severity = suggestion
csharp_indent_braces = false
csharp_new_line_before_open_brace = all

Категории правил анализаторов:

  • CAxxxx — правила качества кода (Code Analysis):
    • Назначение: выявление потенциальных проблем с производительностью, безопасностью, надёжностью, корректностью использования API и архитектурой.
    • Входят в состав .NET SDK, включены по умолчанию для проектов, нацеленных на .NET 5 и выше.
    • Каждое правило имеет свой уровень серьёзности по умолчанию, который можно переопределить в .editorconfig.
    • Примеры правил:
      • CA1805 — избыточная инициализация статических полей значениями по умолчанию.
      • CA2200 — повторное создание исключения без сохранения стека вызовов (throw ex вместо throw).
      • CA1062 — обнаружение потенциального разыменования параметров, допускающих значение null, в публичных API. Правило учитывает контекст nullable-аннотаций и может изменять поведение в зависимости от уровня анализа.
      • CA2007 — рекомендация явно указывать ConfigureAwait(...) в библиотечном и платформенно-независимом асинхронном коде.
      • CA1822 — метод может быть объявлен как статический.
  • IDExxxx — правила стиля и удобства (IDE):
    • Назначение: обеспечение единообразия форматирования, применение современных конструкций языка, повышение читаемости.
    • Работают как анализаторы и как рефакторинги с автоматическим исправлением.
    • Уровни серьёзности по умолчанию: suggestion или silent.
    • Примеры правил:
      • IDE0055 — форматирование кода (отступы, пробелы, расстановка скобок).
      • IDE0001 — упрощение квалифицированного имени типа.
      • IDE0002 — упрощение доступа к членам типа.
      • IDE0017 — предпочтение использования инициализатора объекта.
      • IDE0028 — предпочтение использования инициализатора коллекции.
      • IDE0031 — предпочтение использования null-условного оператора (?.).
      • IDE0063 — упрощённый синтаксис оператора using (без фигурных скобок).
  • CSxxxx — ошибки и предупреждения компилятора C#:
    • Назначение: диагностика синтаксических и семантических ошибок, нарушений типизации, использования небезопасных конструкций.
    • Генерируются непосредственно компилятором Roslyn.
    • Уровни серьёзности:
      • error — ошибки, прерывающие компиляцию (например, CS0103, CS1001).
      • warning — предупреждения, не прерывающие сборку (например, CS0219, CS0649).
    • Примеры правил:
      • CS0103 — имя отсутствует в текущем контексте.
      • CS0219 — переменной присвоено значение, которое не используется.
      • CS0649 — поле не получает значения и будет содержать значение по умолчанию.
      • CS8600 — преобразование null-литерала или потенциально null-значения в ненулевой тип (при включённом nullable-контексте).
      • CS8625 — аргумент со значением null передан в параметр, не допускающий null.

Уровни серьёзности правил:

  • none — правило полностью отключено.
  • silent — правило участвует в анализе и автоматических исправлениях, но обычно не отображается как предупреждение в редакторе (может влиять на code fixes).
  • suggestion — рекомендация по улучшению, не влияющая на успешность сборки.
  • warning — предупреждение, не прерывающее сборку.
  • error — ошибка, приводящая к прерыванию сборки.
  • default — используется уровень серьёзности, заданный по умолчанию для данного правила.

Настройка уровней серьёзности в .editorconfig:

  • Синтаксис записи:
    Код.editorconfig
    dotnet_diagnostic.<RuleId>.severity = <severity>

Правила именования (Naming Rules):

  • Определяются через комбинацию трёх сущностей:
    • Символьная группа (symbols) — элементы, к которым применяется правило (классы, методы, поля, свойства и т.д.).
    • Стиль именования (style) — требуемый формат (PascalCase, camelCase, с префиксами или суффиксами).
    • Уровень серьёзности — реакция на нарушение.
  • Пример настройки правила для публичных членов:
    Код.editorconfig
    # Публичные члены должны использовать PascalCase
dotnet_naming_rule.public_members_should_be_pascal_case.symbols = public_members
dotnet_naming_rule.public_members_should_be_pascal_case.style = pascal_case
dotnet_naming_rule.public_members_should_be_pascal_case.severity = warning

dotnet_naming_symbols.public_members.applicable_kinds = method, property, field
dotnet_naming_symbols.public_members.applicable_accessibilities = public

dotnet_naming_style.pascal_case.capitalization = pascal_case

Настройка стиля кода в .editorconfig:

  • Параметры форматирования (csharp_*):
    • csharp_new_line_before_open_brace — расположение открывающей фигурной скобки.
    • csharp_indent_braces — управление отступами для фигурных скобок.
    • csharp_preserve_single_line_statements — сохранение коротких инструкций в одну строку.
    • csharp_prefer_braces — требование обязательного использования фигурных скобок для конструкций if, else, for.
  • Параметры стиля (dotnet_style_*):
    • dotnet_style_readonly_field — предпочтение модификатора readonly для полей, которые не изменяются.
    • dotnet_style_predefined_type_for_locals_parameters_members — предпочтение встроенных псевдонимов типов (int вместо System.Int32) для локальных переменных, параметров и членов.
    • dotnet_style_object_initializer — предпочтение инициализаторов объектов при создании экземпляров.
    • dotnet_style_explicit_tuple_names — требование явного указания имён элементов кортежа.
  • Языковые параметры (csharp_style_*):
    • csharp_style_pattern_matching — предпочтение конструкций сопоставления с образцом.
    • csharp_style_expression_bodied_members — предпочтение членов, заданных в форме выражения.
    • csharp_style_inlined_variable_declaration — предпочтение встроенного объявления переменных в конструкциях out var.

Управление анализаторами в файле проекта (.csproj):

  • AnalysisLevel — версия набора правил (например, latest, 8.0, 7.0).
  • AnalysisMode — режим анализа: All, Default, Recommended, Minimum, None.
  • TreatWarningsAsErrors — преобразование всех предупреждений в ошибки.
  • WarningsAsErrors — преобразование указанных предупреждений в ошибки.
  • NoWarn — подавление указанных предупреждений.
  • EnableNETAnalyzers — принудительное включение анализаторов .NET для проектов, где они не активны по умолчанию (например, .NET Framework, .NET Core 3.x, .NET Standard 2.0). Для проектов, нацеленных на .NET 5 и выше, анализаторы включены автоматически.

Форматирование кода через CLI:

  • Форматирование всех файлов решения в соответствии с правилами из .editorconfig:
    Терминалbash
    dotnet format MySolution.sln
  • Проверка необходимости форматирования без внесения изменений:
    Терминалbash
    dotnet format --verify-no-changes
  • Форматирование указанного файла:
    Терминалbash
    dotnet format --include MyFile.cs
  • Форматирование только ошибок (игнорирование предупреждений и рекомендаций):
    Терминалbash
    dotnet format --severity error
  • Создание отчёта о результатах форматирования в формате JSON:
    Терминалbash
    dotnet format --report ./report
    • Параметр --report указывает каталог, в котором будет создан файл report.json с результатами форматирования.
  • Дополнительные опции:
    • --diagnostics — фильтрация по идентификаторам правил.
    • --exclude — исключение файлов или каталогов.
    • --include-generated — включение автоматически сгенерированных файлов.
    • --no-restore — пропуск этапа восстановления зависимостей.

Сторонние анализаторы:

  • Устанавливаются в проект через пакеты NuGet или расширения среды разработки.
  • Примеры популярных анализаторов:
    • StyleCop.Analyzers — проверка соответствия стилю кода (правила с префиксом SA).
    • Roslynator — большая коллекция анализаторов и рефакторингов.
    • SonarAnalyzer.CSharp — анализ качества, безопасности и надёжности кода.
    • xunit.analyzers — специализированные анализаторы для тестов на базе xUnit.
    • SecurityCodeScan — выявление потенциальных уязвимостей.
    • AsyncFixer — обнаружение проблем в асинхронном коде.

Подавление предупреждений:

  • Полное отключение правила:
    • Установка severity = none в файле .editorconfig.
  • Подавление для конкретной строки кода с помощью директивы #pragma:
    КодC#
    #pragma warning disable <CheckId>
// Код, вызывающий предупреждение
#pragma warning restore <CheckId>
  • Подавление с помощью атрибута SuppressMessageAttribute:
    КодC#
    [SuppressMessage("Category", "CheckId", Justification = "Обоснование")]
public void MyMethod() { }
  • Глобальный файл подавлений GlobalSuppressions.cs:
    КодC#
    [assembly: SuppressMessage("Design", "CheckId", Scope = "member", Target = "~M:MyClass.MyMethod")]
  • Подавление через интерфейс среды разработки с использованием меню быстрых действий (Quick Actions).

Интеграция с процессами CI/CD:

  • Добавление шага проверки форматирования в конвейер сборки:
    YAML (GitHub Actions)yaml
    - name: Check formatting
  run: dotnet format --verify-no-changes
- name: Build
  run: dotnet build -c Release
  • Параметр --verify-no-changes возвращает ненулевой код завершения, если обнаружены файлы, требующие форматирования. Это приводит к прерыванию конвейера и сигнализирует о необходимости приведения кода к установленным стандартам.

Генерация файла .editorconfig из Visual Studio:

  • Путь в меню: Tools -> Options -> Text Editor -> C# -> Code Style -> General.
  • Настройка параметров стиля через графический интерфейс.
  • Кнопка «Generate .editorconfig file from settings» создаёт файл в корне решения с текущими параметрами.
  • Альтернативный способ: добавление нового элемента в проект через Add -> New Item -> EditorConfig File (.NET).

Создание собственного анализатора:

  • Используется библиотека Microsoft.CodeAnalysis (API компилятора Roslyn).
  • Класс анализатора наследуется от DiagnosticAnalyzer.
  • Определяются метаданные правила через DiagnosticDescriptor (идентификатор, название, сообщение, категория).
  • Реализуется метод Initialize, регистрирующий действия анализа на синтаксические узлы или символы.
  • Готовый анализатор упаковывается в пакет NuGet и подключается к проекту в качестве анализатора.
